Purpose: These are the signature features that make Visionary an intelligent ops center rather than a static dashboard. Interview mode bridges the gap between "I want X" and a well-structured agent task. Jarvis routing removes the need to know which agent handles what.

**server.js — Jarvis routing function (module-level):**

Add a `routeToAgent(description)` function that maps task descriptions to agents using keyword matching. The function takes a string, lowercases it, and checks for keyword matches in this order (first match wins):
- `sentinel`: keywords `security`, `audit`, `vulnerability`, `pentest`, `scan`, `threat`, `cyber`
- `scout`: keywords `news`, `brief`, `morning`, `intelligence`, `scan`, `summary`, `digest`
- `analyst`: keywords `research`, `analysis`, `deep-dive`, `investigate`, `report`, `data`, `compare`
- `forge`: keywords `build`, `code`, `dashboard`, `automate`, `script`, `deploy`, `feature`, `fix`, `bug`
- `broker`: keywords `invest`, `portfolio`, `market`, `stock`, `finance`, `trading`, `crypto`
- `ops`: keywords `infra`, `docker`, `server`, `deploy`, `devops`, `nginx`, `kubernetes`, `backup`
- `hunter`: keywords `job`, `career`, `cv`, `resume`, `linkedin`, `opportunity`, `interview`
- Default: `jarvis` (if no keywords match, Jarvis handles it directly)

The function returns `{ agent_id: string, confidence: 'high'|'medium'|'low', matched_keywords: string[] }`. High confidence = 2+ keyword matches. Medium = 1 match. Low = default to jarvis.

**server.js — Modify POST /api/dispatch:**

**server.js — Interview API endpoints:**

Add these routes:

`POST /api/interview/start` — Body: `{ task_id? }`. If task_id provided, load the task title+description as initial context. Create a new interview_sessions row with initial messages_json containing a system message. The system message instructs Jarvis to interview the user: "You are Jarvis, the chief of staff AI. Interview the user to refine this task. Ask 2-3 clarifying questions about scope, priority, and target outcome. Keep questions short and specific. When you have enough information, summarize the refined task and suggest which agent should handle it." If task_id is set, include the task title/description in context. Dispatch Jarvis via execFile with the system message and return `{ session_id, messages: [{ role: 'assistant', content: jarvis_first_question }] }`. The first Jarvis response should be generated by calling `openclaw agent --agent jarvis --message "<system_prompt>\n\nPlease ask your first clarifying question about this task." --json` with a 30-second timeout. Parse the response text from the CLI output (same parsing as dispatchAgent). Return the session with the first message.

`POST /api/interview/:id/reply` — Body: `{ message }`. Load the session, append user message to messages_json, then call Jarvis with the full conversation context: format all messages as a conversation string and append the user's new message. Call `openclaw agent --agent jarvis --message "<conversation_context>" --json` with 30s timeout. Parse response. Check if Jarvis's response contains "TASK READY:" or "REFINED TASK:" (case-insensitive) — if so, extract the refined title (first line after the marker) and description (remaining text), set suggested_agent via routeToAgent on the description, update session status to 'completed'. Append assistant response to messages_json, update session. Return `{ session_id, messages, status, refined_title?, refined_description?, suggested_agent? }`.

`POST /api/interview/:id/dispatch` — Load the completed session. If session has refined_title and refined_description, update the linked task (or create a new one if no task_id). Set agent_id to suggested_agent or body.agent_id override. Call dispatchAgent(). Return `{ run_id, task_id, agent_id }`.

`GET /api/interview/:id` — Return session with parsed messages.
